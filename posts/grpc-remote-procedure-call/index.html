gRPC – Remote Procedure Call

December 7, 2018 3-minute read
marthinpsrb
technology

Now, i wanna share about gRPC. Before we go to deeply, the first question is , what is it? From the real website of gRPC (grpc.io) the meaning of gRPC:

gRPC is a modern, open source remote procedure call (RPC) framework that can run 
anywhere. It enables client and server applications to communicate transparently, 
and makes it easier to build connected systems.

The next question is, what does gRPC stands for?

gRPC : gRemote Procedure Call

gRPC was found by Google and made open source  in late Feb 2015. gRPC is a CNCF(Cloud Native Computing Foundation) project. Here’s the background: For the modern application, actually now they created frontend and backend. How they communicate?? Yes, using middleware, developer should create API to communicate with frontend. APIs are powering the backend for web client and mobile client applications, and also used for communicate between applications regardless of technology and platform.

When you think about building APIs, you typically choose RESTful APIs along with JSON as the standard for interchanging data between applications. This approach is fine and mobile client applications can easily consume these JSON based RESTful APIs. But now, try to thinking if you using micro services? Do you think it’s good in the performance?

Now we’re building applications for the era of cloud-native applications 
where  our Micro services should be able for massive scale and 
performance is very critical.

Yes, we  need a high performance communication mechanism for communicate between various Microservices. Each services should can communicate with other services. REST, as mentioned earlier, depends heavily on HTTP (usually HTTP 1.1) and the request-response model. On the other hand, gRPC uses the newer HTTP/2 protocol. 

The HTTP/2 protocol provides lot of capability than its previous version,
thus we need to leverage those capabilities when building next-generation APIs.
Then enter to gRPC and protocol buffers.

gRPC enables client and server applications to communicate transparently, and makes it easier to build connected systems.  When you compare between gRPC and REST, from the article that i read (this and this)

By default,

  • gRPC uses Protocol Buffers as the Interface Definition Language (IDL)
  • REST uses JSON

Protocol Buffers are not just message interchange format, it’s also used for describing the service interfaces (service endpoints). Thus Protocol Buffers are used for both the service interface and the structure of the payload (response message by default after client request API ) messages.

The gRPC server implements the service interface and runs an RPC server to handle client calls to its service methods. On the client side, the client has a stub (referred to as just a client in some languages) that provides the same methods as the server.
